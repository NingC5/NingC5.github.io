1. 准备工作

暂未参考：Hexo教程，看这一篇就够了- How to系列-CSDN博客

个人博客是一个 Web 网站，允许用户在浏览器地址栏键入地址来打开。Web 网站有静态和动态之分，静态指用户和网站不能进行交互，动态则相反。

Hexo 是一个快速、简洁且高效的博客框架，它能够解析 Markdown 格式的文档，渲染生成 html 文件。

使用 Hexo + GitHubPages 搭建的个人博客类似于静态 Web 网站，但能实现一些动态效果。

notes：html、js、css 等编程语言的学习比较繁杂，而 Markdown 较为简易 。当然，如果想使网页具有个人色彩，还需对编程语言具有一定的了解。

要使用 Hexo，首先要确保计算机上已经安装了 Node.js 和 Git。Git 是版本控制系统；安装它是为了管理本地博客数据并将博客项目托管到 GitHub。Node.js 是一个运行环境，使得 JavaScript 代码能够在服务器端运行；安装它主要是为了使用其包管理系统 npm（Node Package Manager），所以暂时不需要系统地学习其知识。

2.2 关于博客项目文件夹的说明

Hexo文件结构 - 简书

暂未遇到的问题：hexo博客文件夹下删除了db.json之后，之前的分类（categories）还在？ - 知乎

  • node_modules/：博客项目的模块安装目录。

  • themes/：主题文件夹。Hexo 会根据主题来生成静态页面。

  • _config.landscape.yml：默认主题的配置文件。默认主题包在 node_modules/ 目录下。

  • package.json 和 package-lock.json：博客项目的依赖项配置文件。

  • _config.yml：网站的配置文件。

  • scaffolds/：模版文件夹。hexo 指令新建文章时，Hexo 会根据 scaffold 来创建文件。

  • source/：资源文件夹；用于存放用户资源，比如 Markdown 文档、静态资源等。

    1. source/_posts/：专门用于存放博文的原始 md 文档。
    2. source/images/：自建的图片文件夹。
    3. …

  • public/：hexo g 指令的输出目录，一般不直接操作里面的文件。静态站点的位置，相当于“博客网站根目录”，访问博客就是访问 public 目录下的资源文件。

执行 hexo g 指令时：Hexo 会解析资源文件夹下的 Markdown 和 HTML 文件（以 _开头命名的文件/文件夹和隐藏的文件将会被忽略 ，不包括 _posts 文件夹），渲染得到的 html 文件会输出到 public 文件夹，此外还会在站点根目录下生成一份缓存文件（db.json）；而其他文件会被直接拷贝到 public 文件夹（不包括以 _开头命名的文件/文件夹和隐藏的文件）。

tips：执行 hexo g 指令时，渲染的主要目标就是 source/_posts/ 目录下的 md 文件，生成的 html 文件就是所说的博文。

解释：上节浏览器所展示的 hello-world 博文的原始 md 文档是执行 hexo init 指令时下载的。执行 hexo g 执行时，/source/_posts/hello-world.md 文档被渲染成 hello-world.html 文件并输出到 public 文件夹。

5. 博客的撰写

在站点根目录下执行 hexo new [layout] <title> 命令来创建一篇新文章或新页面。

5.1 布局

layout 是布局（版面）。Hexo 共有三种 layout：post、page 和 draft。 每种布局创建的文件会被保存到不同的路径。

布局 路径
post（文章） source/_posts/
page（页面） source/
draft（草稿） source/_drafts/

使用 new 命令来新建，如果 layout 缺省，则默认使用 _config.yml 配置文件中的 default_layout 参数代替，默认值为 post。

  1. 若 layout 为 post，则会在 source/_posts/ 目录下生成一篇文章——博文（html）的原始文档（md）。
  2. 若 layout 为 page，则会生成 source/title/index.md。
  3. 若 layout 为 draft，则会在 source/_draft/ 目录下生成一篇草稿。

5.2 文件名称（title 部分）

  1. 使用 new 命令来新建文章或草稿时，默认使用 title 作为文件名（前缀部分）。如果 title 包含空格，请使用引号括起来。 如果 title 包含 . 或 / 等符号，会转换成 - 连接。
  2. 使用 new 命令来创建页面时，Hexo 会创建一个以标题为名字的目录，并在目录下放置一个 index.md 文件。

如果要更改默认文件名，可以更改 new_post_name 配置项的值。例如， :year-:month-:day-:title.md 将在文件名前加上创建日期。（没必要，在 “文档结构管理” 章节会提到。）

5.3 草稿

Hexo 默认不会渲染 source/_draft/目录下草稿 md 文档。可使用 hexo publish draft <filename> 命令发表草稿，即将草稿移到到 source/_posts/ 目录下。

当然，也可以在运行 hexo g 命令时添加 --draft 选项或在配置文件中启用 render_draft 设置来渲染草稿。

5.4 文章（模板 & Front-matter）

使用 new 命令来新建时，Hexo 会根据 scaffolds 文件夹内相对应的模板文件来建立文件。一般只需要修改文章（post）的模板文件（scaffolds/post.md）。

例如，执行 hexo new post "test" 命令时，Hexo 会根据 scaffolds/post.md 模板文件的内容建立 md 文档。如下所示：

![源代码模式](使用 Hexo 搭建个人博客/image-20250413131954477.png)

md 文档开头的代码块就是 Front-matter。

Front-matter（前页|前置内容）是文档开头的 YAML 或 JSON 代码块，用于配置文档的各种属性（元信息）。 以 YAML 格式书写时，Front-matter 以三个破折号结束；以 JSON 格式书写时，Front-matter 以三个分号结束。

YAML front matter ——在 Obsidian 笔记中加入属性 - 知乎

Front-matter 常用的设置有：title、date、categories、tags、excerpt、permalink。

  1. title

title 是文章的标题，默认为 <title>（模板中的默认设置）。

  1. categories & tags（分类 & 标签）

只有文章支持分类和标签。 分类按顺序应用于文章，从而形成分类和子分类的层次结构。 标签是在相同的层次结构上定义的，因此它们的出现顺序不重要。

![image-20250413141318068](使用 Hexo 搭建个人博客/image-20250413141318068.png)

如果想应用多个分类层次结构，请使用一个名称列表而不是一个单个名称。 Hexo 会将每个分类视为独立层次结构。

![image-20250413142007086](使用 Hexo 搭建个人博客/image-20250413142007086.png)

  1. excerpt（摘要）

摘要会出现在博客首页，用于对博文进行必要的简介。

方式一：在文档正文使用 <!-- more --> 截断，之前的文字将会被视为摘要。此方式的摘要会出现在博文中。

方式二：在 Front-matter 中指定 excerpt: 值。此方式的摘要不会出现在博文中。

博主按需采用了方式二。博主习惯在文档开头编写文档概述，若摘要内容出现在博文开头显得不美观。

  1. permalink（永久链接）

覆盖文章的永久链接（覆盖配置文件中的该配置）。永久链接应该以 / 或 .html 结尾。

5.5 文档结构管理

当博文越来越多时，其原始 md 文档的管理也成了一个问题。

配置项 new_post_name 不仅与文件名称相关，还涉及文档的目录结构。 例如，将该配置项设置为:year/:month/:title.md，new 命令新建文档时会按年月分类。示例：执行 hexo new post "test2" 命令，Hexo 会生成 source/_posts/2025/4/test2.md。博客需要 多语种支持 时，可以修改该配置项。

私以为上述结构较为死板，当然也可以采用该方式。而本人习惯按类别管理文档。

在运行 new 命令时添加 --path 选项，自定义文档的路径。

例如，执行 hexo new post -p test/test3 "测试文档3" 命令，Hexo 会生成 source/_posts/test/test3.md。

注意！path 部分的最后必须是文件名（可以是前缀名，会自动加 .md 后缀；也可以是带后缀的全名）！后面的 title 也必须指定！否则可能会产生非期望的结果！ 示例的文件名为 test3，title 为 测试文档3，此 title 会作为博文的标题（Front-matter 部分的 title）。

tips：当然，也可以不使用 new 命令和 path 选项。通过直接手动创建目录和文档的方式来达到同样的效果。采用命令需要知晓该命令的执行结果。

6. 文档和博客同时显示图片

Hexo的工作原理探究 | Java技术经验分享

md 文档显示图片很容易实现。

通常采用 Typora 程序编写 md 文档，常使用 md 语法：![](imageLocation) 引用本地图片。

为了方便管理每个 md 文档及其图片，进行偏好设置，【图像】→ 【插入图片时…】→ 【复制到指定路径：./${filename} 】。

相关说明：

  • Typora 当然也支持 html 语法引用图片： <img src="imageLocation"/>，缩放图片时 md 语法就会转换成 html 语法。
  • 为了与 hexo 适配，Typora 设置插入图片时复制到指定路径 ./${filename}，而不是它默认提供的选项：复制图片到 ./${filename}.assets 文件夹。
  • 编写 md 文档时，经常使用截图软件截屏并复制粘贴到 md 文档。如果进行上述设置，Typora 会进行如下操作：
    1. 在文档所在目录下创建与文档同名的文件夹 filename/，并将图片复制到该文件夹下。
    2. 文档中插入该图片，格式为： ![](filename/imageFullName。imageLocation 为图片的相对路径。

访问博客就是访问 public/ 目录下的 html 文件。博客显示图片意味着 【html 文件中图片的引用路径】与【图片在 public/ 目录下的真实路径】 一致。这两者都与 hexo g 指令的执行有关，涉及到网站配置文件（_config.yml）中的如下配置项：

配置 | Hexo

第一步是设置 post_asset_folder: true。 这有两个作用：① 执行 hexo new post <title> 指令时，Hexo 在 source/_posts/ 目录下生成博文的原始 md 文档时，还会同步创建同名的资源文件夹； ② 执行 hexo g 指令时，Hexo 会将文档的资源文件夹复制到 public/ 目录下。（Hexo 复制无法渲染的文件时，默认是忽略 _post/ 目录的。）

然后，聚焦于【执行 hexo g 指令时，Hexo 的默认行为】：

  1. md 文档渲染成 html 文件，输出到【public/ 目录下的 permalink 目录下】。图片的引用路径 【imageLocation】（文档）被渲染成 【/imageLoaction】（博文）。
  2. 复制 md 文档的图片到【public/ 目录下的 permalink 目录下】。

若设置 permalink: :title/，则【html 文件中图片的引用路径】与【图片在 public/ 目录下的真实路径】一致，文档和博文同时显示图片。

【Hexo 的默认处理方式】：Hexo 默认将图片路径渲染成 【/imageLoaction】。这与【图片在 public/ 目录下的真实路径】不一致，因而博文无法显示该图片。

能否正确地获取图片取决于 hexo 对博客项目的解析和输出结果。

为了更好地管理博客项目，博主将资源文件夹分为两个部分。

  1. 第一部分是博客全局资源文件夹。比如：自建的 source/images/ 目录，在里面存放博客的背景图、小图标等。
  2. 第二部分是博文资源文件夹，专供每个 md 文档使用。比如：/source/_post/test.md 文档的资源文件夹 /source/_post/test，在里面存放此文档的图片等资源。
